Corporate Procurement and Manufacturing Inventory SystemIndustrial Context and Operational ChallengesManufacturing enterprises rely on the seamless coordination of physical materials, procurement workflows, and factory schedules to maintain a competitive market position. Inefficient oversight of inventory balances often manifests in two distinct operational bottlenecks: critical raw material stockouts that halt active production lines, and excessive inventory accumulation that ties up working capital in slow-moving items. Historically, small-to-medium manufacturing environments have managed these workflows using isolated spreadsheets. While spreadsheets are accessible, they do not scale, lack native support for relational tracking, and are prone to data entry errors and overwrites.Establishing a centralized, web-based portal bridges the gap between material procurement and production management. Rather than overwriting a single stock balance metric, the system uses a transactional ledger design. This approach maintains accurate real-time inventory balances while capturing the historical timeline of every inbound supplier delivery, raw material consumption event, and completed production run. By providing complete visibility across the material lifecycle, the platform reduces administrative delays, coordinates purchasing with actual production needs, and ensures raw materials are tracked from arrival to final assembly.Core Architectural Design and Concurrency ControlFor early prototyping and validation, the portal uses a file-based JSON storage system, mimicking relational database structures without the immediate overhead of an external SQL engine. While JSON is highly adaptable and simplifies frontend state integration, it presents technical challenges regarding multi-user concurrency and transactional safety on a web server.Concurrency and Transactional Integrity MechanicsBecause web applications handle asynchronous requests, simultaneous writes to a single JSON file risk data loss, race conditions, and file corruption. To mitigate these issues, the system implements an application-level concurrency control pattern:Exclusive Write-Locking: The application handles file access through a custom wrapper. When a request initiates a write operation, the system acquires an exclusive lock on the file. Subsequent read or write requests to that file are queued in memory using a first-in, first-out (FIFO) promise chain.Atomic File Updates: Rather than modifying a JSON file in place, the server loads the dataset, processes modifications in memory, writes the updated payload to a temporary file, and performs an atomic replace operation. This ensures that if a system failure occurs mid-write, the original database file remains uncorrupted.Relational Integrity Mapping: The application layer validates relational logic to maintain clean data structures. Before deleting or modifying a key, the system verifies all linked records across other files, matching the behavior of primary and foreign keys in relational databases.File Database ArchitectureThe file-based storage architecture is organized into seven distinct files, each mapped to a specific operational domain:File NameFunctional PurposePrimary KeyForeign Key LinksKey Schema Attributesusers.jsonStores credentials and authorization roles.userIdNoneusername, passwordHash, role.suppliers.jsonProfiles raw material vendors and lead times.supplierIdNonecompanyName, leadTimeDays, rating.raw_materials.jsonTracks input items, prices, and stock types.materialIdsupplierId[cite: 8, 18]sku, quantityInStock, stockType.finished_goods.jsonLists completed manufactured items.productIdbomId[cite: 19]sku, quantityInStock, unitPrice.bom.jsonDefines assembly bills of materials.bomIdproductIdversion, materials list.purchase_orders.jsonTracks procurement from draft to delivery.poIdsupplierId[cite: 11, 20]status, items array, totalCost.stock_movements.jsonAudits warehouse modifications and reasons.movementIditemId, userId[cite: 6]direction, quantity, reasonCode.Database Schema Definitions and Test DatasetsThe structured JSON schemas and production-grade mock datasets below provide the initial data state for system development and testing.Schema and Seed Dataset: users.jsonTracks authorized employees, system roles, and credentials.JSON[

The Six Stages of Procurement TrackingThe tracking system traces transactions from initial department requisitions to final accounts reconciliation:Purchase Requisition: Staff submit requests listing required materials, quantities, estimated costs, and intended suppliers. These requests remain internal and do not represent a financial commitment to the vendor.Approval: A purchasing manager reviews the request details. If the order meets budget guidelines, it is approved for purchase order generation.Purchase Order Issuance: The system generates a formal PO using details from the approved requisition, assigns a unique tracking number, and sends the document to the supplier. This creates a binding contract and establishes a pending delivery in the system.Physical Receipt Verification: When the delivery arrives at the warehouse, staff count the items and log received quantities against the original PO. Any discrepancies are recorded to maintain inventory accuracy.Three-Way Invoice Matching: To prevent billing errors, the system runs a three-way match comparing the original purchase order, the receiving log, and the vendor's invoice. If the values match within tolerance limits, the invoice is approved for payment.Financial Reconciliation & Closure: The accounts team marks the invoice as paid and shifts the purchase order status to "Closed" in purchase_orders.json, completing the transaction lifecycle.Operational Rules for Inventory IntakeTo ensure inventory accuracy during intake, the system enforces strict data validation rules:Inbound Balance Tracking: Received quantities are verified against the original purchase order. If a delivery is incomplete, the system marks the PO status as "Partially Received" and leaves the remaining balance open. When all items are accounted for, the status is updated to "Received".Quality Hold Isolation: Newly arrived materials default to a "quality_inspection" status in raw_materials.json. This keeps them isolated in the system and prevents them from being assigned to active production runs until quality checks are complete.Transactional Logging: When an intake status is updated, the system increments the material balance in raw_materials.json and logs a confirmation entry in stock_movements.json.Manufacturing Conversions and Bill of Materials ApplicationThe manufacturing module processes physical transformations on the production floor, converting raw input materials into finished goods.+--------------------------------------------------------------------------+

Production Rules and ValidationsTo protect inventory data and prevent stock discrepancies, the system enforces strict validations before any production run can proceed:Stock Status Restrictions: The system only allocates raw materials with an "unrestricted" status to active production runs. Materials marked as "quality_inspection" or "blocked" are bypassed, preventing compromised inputs from entering assembly pipelines.Run Verification: When a user initiates a manufacturing run, the system validates current stock levels against the BOM requirements. If any material falls short of the calculated threshold, the process halts and the user receives an alert detailing the shortage.Synchronized Stock Updates: On successful execution, the system processes updates across related database files. It reduces the raw material balance in raw_materials.json, increases the finished goods count in finished_goods.json, and records the material updates in stock_movements.json.Detailed Movement Reason Technical Constraints and Future RoadmapWhile a file-based prototype is ideal for initial testing, the platform design considers future operational scaling.Operational Boundaries of Flat-File DatabasesBefore moving to a production environment, developers must consider the limitations of JSON storage:Parsing Bottlenecks: Because flat-file databases require the application to read and parse entire JSON strings for every query, system performance can degrade once datasets grow beyond a few thousand rows.No Native Indexing: Without database indexing, queries require sequential scans, which increases CPU load as records accumulate.Single-Server Dependency: File-system locking mechanisms require all web processes to run on a single server, which limits horizontal scaling.Transition to Relational Database Management SystemsTo support future growth, the system's database queries are isolated behind a unified data access layer. This architectural isolation simplifies future database upgrades:+------------------------+
| Application UI Layer   |
+------------------------+
            |
            v
+------------------------+
| Unified Data Interface |
+------------------------+
            |
     +------+------+
     |             |
     v             v
+----------+  +------------+
| JSON DB  |  | RDBMS      |  <-- Future Migration Path
| (Active) |  | PostgreSQL |      (No code changes required
+----------+  +------------+       at the UI layer)
This design ensures that transitioning the data layer to an enterprise database (like SQLite, PostgreSQL, or SQL Server) requires updating only the connection module, leaving the core application logic untouched 1 . This migration strategy allows the platform to scale seamlessly from a prototype to a multi-site enterprise solution 2 .
